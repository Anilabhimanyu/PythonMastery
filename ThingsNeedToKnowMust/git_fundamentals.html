1. Introduction to Git

What is Git?

Git is a distributed version control system created by Linus Torvalds in 2005. It tracks changes in source code during software development and enables multiple developers to work on the same project simultaneously.

Why Git?

  • Distributed: Every developer has a complete copy of the repository
  • Fast: Most operations are local
  • Branching: Lightweight and powerful branching model
  • Data Integrity: Everything is checksummed using SHA-1
  • Collaboration: Seamless team collaboration

Git Installation

# Linux (Debian/Ubuntu)
sudo apt-get install git

# Linux (Fedora)
sudo dnf install git

# macOS (Homebrew)
brew install git

# Windows
# Download from https://git-scm.com/download/win

Initial Configuration

# Set your name and email
git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

# Set default branch name
git config --global init.defaultBranch main

# Set default editor
git config --global core.editor "vim"

# Enable color output
git config --global color.ui auto

# View all settings
git config --list

2. Git Basics

Repository Initialization

# Initialize a new repository
git init

# Clone an existing repository
git clone https://github.com/user/repo.git

# Clone with a different name
git clone https://github.com/user/repo.git my-project

Git File States

Three States of Files:
  • Modified: Changed but not committed
  • Staged: Marked for next commit
  • Committed: Safely stored in local database

Basic Commands

# Check repository status
git status

# Short status format
git status -s

# Add files to staging area
git add filename.py

# Remove from staging area

# Commit changes
git commit -m "Add user authentication"

git commit -am "Update README"

git commit --amend -m "New message"

git log --since="2 weeks ago"

git diff HEAD                # All changes
git diff branch1 branch2     # Between branches

3. Branching

Understanding Branches

Branches are lightweight movable pointers to commits. The default branch is usually called main or master.

Branch Commands

# List branches
git branch                   # Local branches
git branch -r                # Remote branches
git branch -a                # All branches

# Create new branch
git branch feature/user-auth

# Switch to branch
git checkout feature/user-auth

# Create and switch (newer way)
git switch -c feature/user-auth

# Create and switch (older way)
git checkout -b feature/user-auth

# Rename branch
git branch -m old-name new-name

# Delete branch
git branch -d feature/user-auth     # Safe delete
git branch -D feature/user-auth     # Force delete

# Delete remote branch
git push origin --delete feature/user-auth

Branch Best Practices

  • Use descriptive branch names: feature/add-login, bugfix/fix-memory-leak
  • Keep branches short-lived
  • Delete merged branches
  • Regularly sync with main branch

Branch Naming Conventions

feature/feature-name      # New features
bugfix/bug-description    # Bug fixes
hotfix/critical-fix       # Production fixes
release/version-number    # Release branches
docs/documentation-update # Documentation
refactor/code-improvement # Code refactoring

4. Merging & Rebasing

Merging

# Merge branch into current branch
git merge feature/user-auth

# Merge with no fast-forward (creates merge commit)
git merge --no-ff feature/user-auth

# Abort merge
git merge --abort

# Continue after resolving conflicts
git merge --continue

Merge Conflicts

Conflicts occur when Git can't automatically merge changes. You'll need to resolve them manually.

# Example conflict markers
<<<<<<< HEAD
def calculate(x, y):
    return x + y
=======
def calculate(a, b):
    return a + b + 10
>>>>>>> feature/new-calculation

# After resolving conflicts
git add conflicted-file.py
git commit -m "Resolve merge conflict"

Rebasing

Rebasing rewrites commit history by applying commits from one branch onto another.

# Rebase current branch onto main
git rebase main

# Interactive rebase (last 3 commits)
git rebase -i HEAD~3

# Continue after resolving conflicts
git rebase --continue

# Skip a commit during rebase
git rebase --skip

# Abort rebase
git rebase --abort

Interactive Rebase Commands

pick   = use commit
reword = use commit, but edit message
edit   = use commit, but stop for amending
squash = use commit, meld into previous commit
fixup  = like squash, but discard message
drop   = remove commit
Warning: Never rebase commits that have been pushed to a public repository and others are working on!

Merge vs Rebase

Merge Rebase
Preserves complete history Creates linear history
Non-destructive Rewrites history
Creates merge commits No merge commits
Safe for public branches Dangerous for public branches

5. Working with Remote Repositories

Remote Commands

# List remotes
git remote -v

# Add remote
git remote add origin https://github.com/user/repo.git

# Change remote URL
git remote set-url origin https://github.com/user/new-repo.git

# Remove remote
git remote remove origin

# Rename remote
git remote rename origin upstream

Fetching and Pulling

# Fetch from remote (doesn't merge)
git fetch origin

# Fetch all remotes
git fetch --all

# Pull (fetch + merge)
git pull origin main

# Pull with rebase
git pull --rebase origin main

# Pull specific branch
git pull origin feature/user-auth

Pushing

# Push to remote
git push origin main

# Push new branch and set upstream
git push -u origin feature/user-auth

# Push all branches
git push --all

# Push tags
git push --tags

# Force push (dangerous!)
git push --force
git push --force-with-lease  # Safer alternative

# Delete remote branch
git push origin --delete feature/old-feature

Tracking Branches

# Set upstream branch
git branch --set-upstream-to=origin/main main

# Push and set upstream
git push -u origin feature/new-feature

# View tracking branches
git branch -vv

6. Advanced Git Concepts

Stashing

Stash allows you to save uncommitted changes temporarily.

# Stash changes
git stash

# Stash with message
git stash save "WIP: working on authentication"

# List stashes
git stash list

# Apply most recent stash
git stash apply

# Apply specific stash
git stash apply stash@{2}

# Apply and remove from stash list
git stash pop

# Drop stash
git stash drop stash@{0}

# Clear all stashes
git stash clear

# Show stash contents
git stash show -p stash@{0}

Cherry-picking

Apply specific commits from one branch to another.

# Cherry-pick a commit
git cherry-pick 

# Cherry-pick multiple commits
git cherry-pick  

# Cherry-pick a range
git cherry-pick ..

# Cherry-pick without committing
git cherry-pick -n

Reverting and Resetting

# Revert a commit (creates new commit)
git revert 

# Revert without committing
git revert -n 

# Reset to specific commit (soft - keeps changes staged)
git reset --soft 

# Reset (mixed - keeps changes unstaged, default)
git reset --mixed 

# Reset (hard - discards all changes)
git reset --hard 

# Reset to previous commit
git reset --hard HEAD~1

# Reset specific file
git checkout -- filename.py
Danger: git reset --hard permanently deletes uncommitted changes!

Reflog

Reflog tracks all changes to HEAD, allowing recovery of "lost" commits.

# View reflog
git reflog

# Recover lost commit
git reset --hard 

# View reflog for specific branch
git reflog show feature/user-auth

Tags

# List tags
git tag

# Create lightweight tag
git tag v1.0.0

# Create annotated tag
git tag -a v1.0.0 -m "Release version 1.0.0"

# Tag specific commit
git tag -a v1.0.0  -m "Release 1.0.0"

# Show tag information
git show v1.0.0

# Push tags
git push origin v1.0.0
git push origin --tags

# Delete local tag
git tag -d v1.0.0

# Delete remote tag
git push origin --delete v1.0.0

# Checkout tag
git checkout v1.0.0

Git Bisect

Binary search to find the commit that introduced a bug.

# Start bisect
git bisect start

# Mark current commit as bad
git bisect bad

# Mark a known good commit
git bisect good 

# Git will checkout a commit in between
# Test and mark as good or bad
git bisect good  # or git bisect bad

# Repeat until bug is found
# Reset when done
git bisect reset

Git Hooks

Scripts that run automatically on certain Git events.

Hooks are stored in .git/hooks/

Common hooks: pre-commit, pre-push, post-merge

# Example pre-commit hook (.git/hooks/pre-commit)
#!/bin/sh
# Run Python linting before commit

python -m pylint **/*.py
if [ $? -ne 0 ]; then
    echo "Linting failed. Commit aborted."
    exit 1
fi

Submodules

# Add submodule
git submodule add https://github.com/user/library.git lib/library

# Clone repo with submodules
git clone --recursive https://github.com/user/repo.git

# Initialize submodules
git submodule init
git submodule update

# Update all submodules
git submodule update --remote

# Remove submodule
git submodule deinit lib/library
git rm lib/library

Git Worktrees

Work on multiple branches simultaneously without switching.

# Create new worktree
git worktree add ../project-feature1 feature/feature1

# List worktrees
git worktree list

# Remove worktree
git worktree remove ../project-feature1

# Prune deleted worktrees
git worktree prune

7. Git Workflows

Git Flow

A popular branching model with strict branch structure.

Branches:
  • main: Production-ready code
  • develop: Integration branch for features
  • feature/*: New features
  • release/*: Release preparation
  • hotfix/*: Emergency production fixes
# Feature workflow
git checkout develop
git checkout -b feature/new-feature
# ... work on feature ...
git checkout develop
git merge --no-ff feature/new-feature
git branch -d feature/new-feature

# Release workflow
git checkout develop
git checkout -b release/1.0.0
# ... prepare release ...
git checkout main
git merge --no-ff release/1.0.0
git tag -a v1.0.0
git checkout develop
git merge --no-ff release/1.0.0
git branch -d release/1.0.0

# Hotfix workflow
git checkout main
git checkout -b hotfix/critical-bug
# ... fix bug ...
git checkout main
git merge --no-ff hotfix/critical-bug
git tag -a v1.0.1
git checkout develop
git merge --no-ff hotfix/critical-bug
git branch -d hotfix/critical-bug

GitHub Flow

Simpler workflow focused on continuous deployment.

Process:
  1. Create a branch from main
  2. Make changes and commit
  3. Open a Pull Request
  4. Discuss and review code
  5. Deploy for testing
  6. Merge to main

Trunk-Based Development

Developers work on short-lived branches or directly on trunk (main).

  • Commit directly to main or use very short-lived branches (1-2 days)
  • Requires strong CI/CD and testing
  • Use feature flags for incomplete features

Forking Workflow

Common in open-source projects.

# Fork repository on GitHub
# Clone your fork
git clone https://github.com/your-username/repo.git

# Add upstream remote
git remote add upstream https://github.com/original-owner/repo.git

# Keep fork synced
git fetch upstream
git checkout main
git merge upstream/main
git push origin main

# Create feature branch
git checkout -b feature/new-feature

# Push to your fork
git push origin feature/new-feature

# Create Pull Request from your fork to upstream

8. Real-Time Issues & Scenarios

Scenario 1: Accidentally Committed to Wrong Branch

Problem: You made commits on main instead of a feature branch.

# Create new branch with current commits
git branch feature/my-feature

# Reset main to origin
git reset --hard origin/main

# Switch to feature branch
git checkout feature/my-feature

Scenario 2: Undo Last Commit (Keep Changes)

# Undo commit but keep changes staged
git reset --soft HEAD~1

# Undo commit and unstage changes
git reset HEAD~1

# Modify and recommit
git add .
git commit -m "Correct commit message"

Scenario 3: Accidentally Deleted Branch

# Find the commit hash in reflog
git reflog

# Recreate branch
git checkout -b feature/recovered

Scenario 4: Merge Conflict Resolution

# Start merge
git merge feature/user-auth

# Conflict occurs - edit files manually
# Remove conflict markers and decide which code to keep

# After resolving
git add resolved-file.py
git commit

# Or use merge tools
git mergetool

Scenario 5: Committed Sensitive Data (passwords, API keys)

# Remove file from last commit
git rm --cached config/secrets.py
git commit --amend -m "Remove sensitive data"

# If already pushed, use BFG Repo-Cleaner or git-filter-repo
# BFG: https://rtyley.github.io/bfg-repo-cleaner/
java -jar bfg.jar --delete-files secrets.py

# Force push (coordinate with team!)
git push --force
Important: Rotate any exposed secrets immediately! Removing from Git history doesn't remove from GitHub history or clones.

Scenario 6: Pull Latest Changes with Local Modifications

# Option 1: Stash, pull, apply
git stash
git pull origin main
git stash pop

# Option 2: Commit, pull with rebase
git commit -am "WIP"
git pull --rebase origin main

# Option 3: Pull and merge
git pull origin main

Scenario 7: Sync Fork with Upstream

# Add upstream if not already added
git remote add upstream https://github.com/original/repo.git

# Fetch upstream
git fetch upstream

# Merge upstream main into your main
git checkout main
git merge upstream/main

# Push to your fork
git push origin main
